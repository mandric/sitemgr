# Design Review — 2026-03-03

Critical review of vision.md, architecture.md, interfaces.md, and
platform-apis.md.

---

## What's strong

**The problem is real and clearly articulated.** The "metadata gap" framing —
your photos are dumb files with no searchable content — is the strongest hook.
The bed repair walkthrough is concrete and compelling.

**The provider abstraction is clean.** Three BYO contracts (storage,
enrichment, query), each with a small surface area. Pluggable without being
over-engineered.

**SQLite per-device is a good call.** Eliminates an entire class of distributed
systems problems. WAL mode, FTS5 co-located with the data, content-addressed
blobs — boring, correct choices.

**The "what we're NOT building" section is unusually good.** That list should
be treated as sacred.

---

## What concerns me

### 1. The "agent-first" bet is premature

The entire UX story is: talk to an AI agent, it calls the query API. But for
v0 targeting developers, this is backwards. Developers want a solid CLI they
can script, pipe, and debug. They'll use the agent *after* they trust the data
layer.

What happens when the agent gets confused? There's no fallback UI for
browsing, filtering, or managing data. If enrichment tagged something wrong,
how do you correct it?

**Recommendation:** CLI-first, agent-second. The agent is a power feature,
not the foundation.

### 2. Enrichment costs are unmodeled

10-20 photos/day = 300-600 enrichment calls/month. At $0.01-0.05/call that's
$3-30/month — not trivial for a "no subscription" tool. There's no cost
estimation, budget tracking, or model selection guidance.

### 3. Project context doesn't work with single-item enrichment

The vision says "the LLM recognizes the ongoing project context across
photos" but the enrichment interface is single-item. Each photo is enriched
in isolation with no mechanism for cross-photo context.

Without batch/session enrichment, you're relying on the LLM to independently
tag each photo with "bed-repair" — which it won't reliably do. Photo 1 gets
"furniture damage," photo 5 gets "woodworking," photo 8 gets "home
improvement."

**Recommendation:** Either scope the v0 walkthrough to independent queries
("find all photos of wood") or promote batch enrichment to Phase 1. A simple
heuristic — "enrich photos taken within 30 minutes together" — would help.

### 4. Android background execution is unresolved

"Foreground service or WorkManager" are fundamentally different:
- Foreground service: persistent notification, battery drain, Android 12+
  restrictions on starting them
- WorkManager: batched, deferred, no timeliness guarantee

"Within seconds, an event is logged" requires a foreground service. This
needs a concrete decision and real-device testing, not a deferred choice.

### 5. Multi-device access is central but undesigned

"Cross-device sync is a future concern" but the primary user flow is: take
photo on phone, query from laptop. How does the phone's events.db get to the
laptop?

**Recommendation:** For v0, the simplest path: periodic SQLite export to S3.
Desktop CLI downloads and ATTACHes it. Ugly but functional, validates the
architecture.

### 6. Git sync has a landmine

> "Conflict resolution assumed solved (CRDTs, last-write-wins, or similar)"

Git conflicts are explicitly NOT solved. If two devices edit the same note,
`git push` fails. Then what?

**Recommendation:** For v0, make document sync single-writer. Each device
pushes to its own branch or directory. No merge. Consistent with per-device
event store model.

### 7. Blob storage triple-copies everything

Photos exist in three places: original (DCIM), blob cache (~/.sitemgr/blobs),
S3 remote. For videos (100MB+) this is a storage problem. No discussion of
cache eviction or local storage pressure.

### 8. Event store grows forever

Append-only with no compaction, archival, or lifecycle story. A year of
active use = 50,000+ events. Users will ask "my events.db is 500MB, is that
normal?" and the answer should be designed, not accidental.

### 9. Enrichment schema is photo-centric

The EnrichmentResult struct (description, objects, context, suggested_tags)
works for photos of physical things. Awkward for screenshots, receipts,
diagrams, memes. Consider content-type-aware prompting or a more flexible
schema.

### 10. Phase 1 scope is large

Phase 1 includes: Android app, SQLite, enrichment + Anthropic, async
pipeline, S3 storage, FTS5, query provider, AND a basic UI. That's a full
product.

**Recommendation:** Split Phase 1:
- **1a:** Android app + SQLite + S3 sync. No enrichment. Prove capture →
  store → sync works reliably with Android background constraints.
- **1b:** Add enrichment on top of a working pipeline.

---

## Smaller items

- **No observability.** Multiple async pipelines with no logging, health
  checks, or debugging story. "Why wasn't my photo enriched?" needs an answer.
- **No HTTP API auth.** "Runs on localhost" works now but Phase 4 web
  dashboard needs auth.
- **No enrichment schema versioning.** If EnrichmentResult changes, old
  events have stale data. raw_response helps but no migration story.
- **`smgr://` custom URI adds friction.** Every consumer must implement
  resolution. Standard URIs + lookup table may be simpler.
- **Name:** "sitemgr" sounds like a website management tool. From the
  candidates, Logbook or Trunk communicate the concept better.

---

## Bottom line

The core insight is strong: capture everything, enrich with LLMs, query with
natural language. The architecture is well-layered with good technology
choices. Main risks: (1) enrichment doesn't produce project-level context
the walkthrough depends on, (2) Android background execution is unresolved,
(3) multi-device access is essential but undesigned.

Recommendation: tighten Phase 1 to capture → sync on Android (no enrichment),
prove that works, then layer enrichment on top. Flip from agent-first to
CLI-first — developers trust tools they can see and script.
