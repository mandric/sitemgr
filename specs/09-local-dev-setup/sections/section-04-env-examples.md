# Section 04: Fix `.env.example` Files

## Overview

Both `.env.example` (root) and `web/.env.example` are reference files that developers copy when first configuring their environment. They currently contain stale or incomplete information:

- Root `.env.example` uses the deprecated `ENCRYPTION_KEY=` name (removed; replaced by `ENCRYPTION_KEY_CURRENT`).
- `web/.env.example` is minimal — it is missing S3 vars, the encryption key, and the service role key.

When these files are wrong, developers configure wrong variable names and encounter confusing runtime errors.

This section has no dependencies on other sections and can be implemented in parallel with sections 01, 03, 05, 06, and 09.

1. **Added `DATABASE_URL=` to `web/.env.example`**: The plan's variable list in the section spec omitted `DATABASE_URL`, but `local-dev.sh print_setup_env_vars` outputs it. Added under the Supabase section so the file accurately documents all vars the script produces.

## Key Decisions

- `ENCRYPTION_KEY_PREVIOUS=` is included as a commented line (not omitted) so developers are aware it exists during key rotation. They should not uncomment it unless actively rotating keys.
- `SMGR_S3_BUCKET=media` and `SMGR_DEVICE_ID=local-dev` ship with sensible local defaults so copying the file requires fewer changes for local dev.
- `SMGR_AUTO_ENRICH=false` defaults to off to avoid accidental Anthropic API calls for new developers.
- `SUPABASE_SECRET_KEY` stays in the Supabase section (not a separate "server" section) for grouping consistency with the root `.env.example`.
- The `web/.env.example` does not include deploy-time vars (`SUPABASE_ACCESS_TOKEN`, `VERCEL_TOKEN`, etc.) — those belong only in the root `.env.example` which documents CI/CD secrets.
