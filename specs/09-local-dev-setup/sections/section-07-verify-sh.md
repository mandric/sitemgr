# Section 7: Add `scripts/setup/verify.sh`

## Overview

This section creates a new shell script at `scripts/setup/verify.sh` that gives immediate, component-level feedback on environment health after running `local-dev.sh`. It is the acceptance test for the whole local dev setup flow.

## Dependencies

- **section-01-local-dev-sh** must be completed first. The `verify.sh` script assumes `.env.local` has been generated by `scripts/local-dev.sh print_setup_env_vars > .env.local`. Specifically it expects `SMGR_API_URL`, `SMGR_API_KEY`, `SUPABASE_SECRET_KEY`, `ENCRYPTION_KEY_CURRENT`, `AWS_ACCESS_KEY_ID`, and `AWS_SECRET_ACCESS_KEY` to be present in `.env.local`.

## File to Create

**`scripts/setup/verify.sh`** — new file; the `scripts/setup/` directory may need to be created.

## Background

After running `local-dev.sh`, a developer has no way to quickly confirm the environment is healthy without running the full test suite. This script provides immediate, component-level feedback.

The script must be callable directly (without the developer having manually `source`d `.env.local`). It sources the file itself before running checks. Each check is wrapped in a function so a single check failure doesn't stop remaining checks — the developer sees all failures at once.

## Expected Output Format

Implement to match this exact output format first; the implementation should produce this output:

```
# When all checks pass:
  ✓ SMGR_API_URL is set
  ✓ SMGR_API_KEY is set
  ✓ SUPABASE_SECRET_KEY is set
  ✓ ENCRYPTION_KEY_CURRENT is set
  ✓ AWS_ACCESS_KEY_ID is set
  ✓ AWS_SECRET_ACCESS_KEY is set
  ✓ Supabase API reachable
  All checks passed.

# When one or more checks fail:
  ✗ SMGR_API_URL is set: variable is empty or unset
  ✗ Supabase API reachable: curl returned non-200
  2 check(s) failed.
```

Each check line is indented with two spaces. Pass lines use `✓`, fail lines use `✗` followed by a colon and the failure reason.

## Tests (Manual Verification)

These verification commands should be run after implementation to confirm correct behavior:

```bash
# Test: exits 0 when Supabase running and vars set
./scripts/local-dev.sh print_setup_env_vars > .env.local
./scripts/setup/verify.sh; echo "exit: $?"  # should be 0

# Test: exits 1 and names the missing var when a var is unset
env -i ./scripts/setup/verify.sh 2>&1 | grep "SMGR_API_URL"

# Test: exits 1 with clear message when Supabase not reachable
supabase stop
./scripts/setup/verify.sh 2>&1 | grep "Supabase API"
supabase start

# Test: works without manually sourcing .env.local first
# (script sources the file itself)
unset SMGR_API_URL
./scripts/setup/verify.sh  # should still pass if .env.local exists
```

## Implementation Details

### Script structure

The script follows this overall structure:

```
set -euo pipefail

# Source .env.local if it exists (so script works without manual sourcing)
if [ -f ".env.local" ]; then
  set -a
  source .env.local
  set +a
fi

failures=0

# Helper functions: check_var, check_api_reachable
# Each function increments $failures on failure, never exits early

# Run all checks
check_var "SMGR_API_URL"
check_var "SMGR_API_KEY"
check_var "SUPABASE_SECRET_KEY"
check_var "ENCRYPTION_KEY_CURRENT"
check_var "AWS_ACCESS_KEY_ID"
check_var "AWS_SECRET_ACCESS_KEY"
check_api_reachable

# Final summary
if [ "$failures" -eq 0 ]; then
  echo "  All checks passed."
  exit 0
else
  echo "  ${failures} check(s) failed."
  exit 1
fi
```

### `check_var` function

- Takes a variable name as argument (not a value — use indirect expansion: `${!var_name}`)
- If the variable is non-empty: print `  ✓ <var_name> is set`
- If empty or unset: print `  ✗ <var_name> is set: variable is empty or unset`; increment `failures`

### `check_api_reachable` function

- Runs: `curl -sf "$SMGR_API_URL/rest/v1/" -H "apikey: $SMGR_API_KEY"`
- The `-s` flag silences curl output; `-f` causes non-zero exit on HTTP error
- If the curl succeeds (exit 0): print `  ✓ Supabase API reachable`
- If the curl fails: print `  ✗ Supabase API reachable: curl returned non-200`; increment `failures`
- Note: `SMGR_API_URL` may be unset if the var check already failed. Guard against this: only run the curl if `SMGR_API_URL` is non-empty, otherwise print `  ✗ Supabase API reachable: SMGR_API_URL not set` and increment `failures`.

### Script file permissions

The script must be executable. After creating the file, run:
```bash
chmod +x scripts/setup/verify.sh
```

### Interaction with `set -euo pipefail`

The `set -euo pipefail` at the top would cause the script to exit immediately if any check function returns non-zero. To prevent this, use `|| true` on check function calls, or use a local `return` pattern inside each check function so the function always returns 0 (it increments a counter rather than returning non-zero on failure). The `failures` variable tracks state; only the final summary exits non-zero.

## Integration with the Setup Flow

This script is called as the third step in the local setup flow (documented in section-08-docs-readme):

```
./scripts/setup.sh
./scripts/local-dev.sh print_setup_env_vars > .env.local
./scripts/setup/verify.sh   # all checks pass
./scripts/test-integration.sh --skip-ollama
```

Running `verify.sh` provides fast confirmation before committing to the full integration test run. It is also the first thing to run during troubleshooting: "Run `./scripts/setup/verify.sh` first. Then `supabase logs`."
