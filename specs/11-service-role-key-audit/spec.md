# 11-service-role-key-audit — Spec

## Overview

Audit the workaround in commit `82b268d` that manually generates an ES256 JWT for `SUPABASE_SECRET_KEY` in `scripts/local-dev.sh`. Determine whether this is the correct approach, or whether the codebase is misusing the service role key in ways that require a different fix.

## Background

Supabase CLI ≥ 2.78 switched GoTrue from HS256 to ES256 JWT signing. The `service_role_key` output by `supabase status -o json` is still an HS256 JWT. When GoTrue is configured with `GOTRUE_JWT_KEYS` (EC key pair), it rejects HS256 tokens with `"signing method HS256 is invalid"`.

Commit `82b268d` works around this by:
1. Inspecting the `supabase_auth_*` Docker container for `GOTRUE_JWT_KEYS`
2. Extracting the EC private key
3. Signing a hand-crafted JWT: `{iss: "supabase-local", role: "service_role", exp: 9999999999}`
4. Using that JWT as `SUPABASE_SECRET_KEY` in `.env.local`

## The Question

**Is manually signing JWTs with GoTrue's private key the right approach?**

The user's intuition: "Shouldn't the JWT get built or generated by the auth service? I don't think a JWT will ever be needed by the service role, only for users."

## Analysis

### How Supabase service_role_key actually works

In Supabase, the `service_role_key` **is** a JWT — not an opaque API key. It's a pre-signed token with:

```json
{
  "role": "service_role",
  "iss": "supabase",
  "exp": <far future>
}
```

This JWT is verified by **PostgREST** (the API layer), not GoTrue (the auth service). PostgREST checks the JWT signature and maps the `role` claim to a Postgres role that bypasses RLS. So the service role key is a legitimate JWT — that part is expected.

### Where the confusion arises

The commit conflates two different verification boundaries:

| Component | Verifies JWTs? | Accepts service_role JWT? |
|-----------|---------------|--------------------------|
| **PostgREST** (API gateway) | Yes — validates JWT signature, extracts `role` | Yes — maps `service_role` to Postgres role, bypasses RLS |
| **GoTrue** (auth service) | Yes — validates JWTs on admin endpoints | **Only for user management admin API** (e.g., `auth.admin.createUser()`) |
| **Storage API** | Yes — validates JWT for bucket/object operations | Yes — `Authorization: Bearer <service_role_key>` |

The HS256→ES256 breakage only affects calls that go through **GoTrue's admin API** (user CRUD). PostgREST and Storage may still be using the HS256 JWT secret, depending on the local Supabase configuration.

### What this codebase uses SUPABASE_SECRET_KEY for

| Usage | File | What it does |
|-------|------|-------------|
| **Admin Supabase client** | `web/lib/media/db.ts:61-65` | `createClient(url, serviceKey)` — calls go to PostgREST |
| **CLI client** | `web/bin/smgr.ts:49` | Same admin client, reads `SUPABASE_SECRET_KEY` env var |
| **Integration test setup** | `web/__tests__/integration/setup.ts:11` | Creates admin client for test seeding |
| **CI bucket creation** | `.github/workflows/ci.yml:148` | `Authorization: Bearer $SUPABASE_SECRET_KEY` to Storage API |
| **Production bucket creation** | `.github/workflows/ci.yml:261` | Same, against production Supabase |
| **Instrumentation** | `web/instrumentation.ts:12` | Validates env var exists at startup |

### What's actually broken without the workaround?

Need to determine: which of the above calls actually fail with the HS256 key on CLI ≥ 2.78?

- **PostgREST calls** (most usage): May still work if PostgREST uses the old JWT secret. Need to verify.
- **GoTrue admin calls**: Will fail — GoTrue only accepts ES256 now.
- **Storage calls**: May use the same signing config as GoTrue. Need to verify.

### Is this codebase calling GoTrue admin API?

Search results show the app uses `supabase.auth.admin.createUser()` and `supabase.auth.admin.deleteUser()` only in **integration test setup** (`setup.ts`). The production app does not call GoTrue admin APIs with the service role key — it uses PostgREST for all data operations.

**This means the ES256 workaround may only be needed for integration tests, not for the app itself.**

## Issues with the current workaround

### 1. Reaching into Docker internals is fragile

Extracting `GOTRUE_JWT_KEYS` from the auth container's environment is an undocumented approach. It breaks if:
- Container naming convention changes
- GoTrue changes how it stores keys
- Docker is not accessible (rootless, remote)

### 2. Hand-crafted JWT may be incomplete

The generated JWT has minimal claims: `{iss, role, exp}`. The real service_role_key may include additional claims (`sub`, `aud`, `iat`) that some endpoints expect.

### 3. Naming inconsistency compounds the problem

The codebase uses two names for the same value:
- `SUPABASE_SECRET_KEY` — in env files, CLI, instrumentation
- `SUPABASE_SERVICE_ROLE_KEY` — in server code (`health/route.ts`, `agent/core.ts`)

This dual naming makes it hard to reason about which key goes where.

## Proposed investigation

### Step 1: Determine what Supabase CLI ≥ 2.78 actually outputs

Check whether `supabase status -o json` on CLI ≥ 2.78:
- Still outputs an HS256 `SERVICE_ROLE_KEY` (likely a bug/gap in the CLI)
- Outputs an ES256 key (meaning the workaround is unnecessary)
- Has a new field for the ES256 key

**Action:** Check the [Supabase CLI changelog](https://github.com/supabase/cli/releases) and issues for ES256 migration notes.

### Step 2: Determine which endpoints actually reject HS256

Run the integration tests with the **unmodified** HS256 `service_role_key` from `supabase status` and log exactly which API calls fail:
- PostgREST queries (`.from("events").select(...)`)
- GoTrue admin calls (`auth.admin.createUser()`)
- Storage API calls (`storage.from("media").upload(...)`)

### Step 3: Evaluate alternatives

| Option | Description | Pros | Cons |
|--------|-------------|------|------|
| **A. Current workaround** | Generate ES256 JWT in local-dev.sh | Works now | Fragile, undocumented, reaches into Docker internals |
| **B. Pin Supabase CLI version** | Lock to < 2.78 in CI and local dev | Simple | Blocks upgrades, temporary |
| **C. Use `supabase status` ES256 key** | If CLI provides one, use it | Clean, official | May not exist yet |
| **D. Separate keys per service** | Use HS256 key for PostgREST, ES256 for GoTrue admin | Correct per-service auth | More env vars, more complexity |
| **E. Avoid GoTrue admin API** | Use SQL/PostgREST for user setup in tests instead of `auth.admin.*` | Eliminates the problem | May not be possible for all auth operations |
| **F. Report upstream** | File issue on supabase/cli requesting ES256 service_role_key in `supabase status` output | Fixes root cause | Depends on upstream timeline |

### Step 4: Unify naming

Regardless of the key approach, consolidate to a single env var name. Candidates:
- `SUPABASE_SERVICE_ROLE_KEY` (more descriptive, matches Supabase dashboard naming)
- `SUPABASE_SECRET_KEY` (shorter, current env file convention)

Pick one and use it everywhere.

## Recommended approach

1. **File upstream issue** (option F) — the CLI should output a working service_role_key for the signing algorithm it configures
2. **Narrow the workaround** — if only GoTrue admin calls break, limit the ES256 JWT to integration test setup only, not the entire `SUPABASE_SECRET_KEY`
3. **Unify naming** to `SUPABASE_SERVICE_ROLE_KEY` across all code
4. **Document** which Supabase services the service_role_key is used with and why
